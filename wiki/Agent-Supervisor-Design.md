## Problem Statement

Racc is an agentic IDE that orchestrates multiple AI coding agent sessions. It already has the core plumbing — PTY transport (local + SSH/tmux), git worktrees, headless server mode, and a React UI. But two critical gaps prevent it from becoming a daily-driver tool for its target users (solo devs with beefy servers, power users tired of SSH+tmux):

1. **Setup friction**: Getting Racc running on a remote server requires too many manual steps.
2. **Cognitive overload**: Spawning agents is easy, but managing many concurrent agents creates mental burden. The more agents you run, the harder it is to track what each is doing, which are stuck, and what needs attention.

The second gap is the deeper insight — it's a fundamental tension in multi-agent tooling. If the tool's value proposition is "run many agents," but more agents = more cognitive load, the tool defeats its own purpose at scale.

## What Makes This Cool

A K8s-inspired supervisor pattern that manages agent lifecycle autonomously. You declare what you want done ("fix these 3 bugs, add this feature"), and Racc's reconciliation loop handles:
- **Assignment**: Which agent gets which task
- **Health monitoring**: Detecting stuck, failed, or idle agents
- **Self-healing**: Auto-restarting failed agents, reassigning stuck tasks
- **Escalation**: Notifying you only when human judgment is needed

The "whoa" moment: you define 5 tasks, walk away, come back to find 4 completed and 1 waiting for your input — with a clear explanation of why it needs you.

## Constraints

- Must build on existing PTY + transport architecture (no rewrites)
- Must work in both Tauri desktop and headless server modes
- Agent-agnostic: must work with Claude Code, Aider, Codex, or any terminal agent
- Solo dev focus: optimize for 1 person managing 3-10 agents, not team-scale
- Open source: architecture should be understandable and extensible by contributors

## Premises

1. The core differentiator is "manage many agents without cognitive overload" — not just "run agents remotely"
2. The next phase should shift from interactive terminal UX toward a more declarative, fire-and-forget model (like CI/K8s)
3. Setup simplification is necessary but not sufficient — it enables adoption but doesn't drive retention
4. The existing PTY + transport architecture is solid enough to build on

## Approaches Considered

### Approach A: Task Queue + Status Dashboard (Minimal Viable)
Add a task queue where you define what needs doing, agents pick up work, and you get a K8s-style dashboard showing health/progress/completion. Effort: M. Risk: Low. Builds directly on existing models but doesn't fully solve the "fire-and-forget" vision. Still requires manual agent assignment.

### Approach B: Declarative Agent Workflows (Ideal Architecture)
GitHub Actions-style YAML workflow definitions with dependency graphs, automatic sequencing, and full observability. Effort: L. Risk: Medium. Most complete long-term but risks over-engineering the YAML schema before validating the UX.

### Approach C: Agent Supervisor Pattern (Creative/Lateral)
K8s controller-inspired reconciliation loop. Declare desired state ("I want these tasks done"), supervisor manages agent lifecycle — auto-assigns, detects stuck agents, restarts failures. You interact with intent, not individual agents. Effort: M-L. Risk: Medium.

## Recommended Approach

**Approach C: Agent Supervisor Pattern** — chosen because it directly addresses the core insight (cognitive overload) and maps naturally to the K8s + GitHub Actions mental model the project aspires to.

### Architecture

```
User Intent (tasks)
       |
       v
  +-----------+
  | Supervisor |  <-- Reconciliation loop (runs every N seconds)
  +-----------+
       |
       +-- Desired State: list of tasks with priorities + constraints
       +-- Actual State:  agent sessions, their status, current work
       +-- Diff:          what needs to change
       |
       v
  +-------------------+
  | Agent Pool Manager |
  +-------------------+
       |
       +-- Spawn new agents when tasks > available agents
       +-- Assign tasks to idle agents
       +-- Detect stuck agents (no output for N minutes, error patterns)
       +-- Restart failed agents (with exponential backoff)
       +-- Kill idle agents when no work remains
       |
       v
  +------------------+
  | Notification Hub  |
  +------------------+
       +-- Task completed (success/failure)
       +-- Agent needs human input (stuck on permission, unclear task)
       +-- Budget/token limit approaching
       +-- All work done
```

### Core Components to Build

1. **Supervisor Service** (`racc-core/src/supervisor.rs`)
   - Reconciliation loop running every **5 seconds** (default). 5s balances responsiveness with DB/lock contention — fast enough to catch stuck agents within ~10s, slow enough to avoid thrashing under load. Configurable via `RACC_SUPERVISOR_INTERVAL_MS`.
   - Desired state vs actual state diffing
   - Decision engine: assign, restart, escalate, or kill
   - State machine: `Pending -> Assigned -> Running -> Completed | Failed | NeedsInput`
   - **Schema migration**: The existing `task.rs` validates statuses as `["open", "working", "closed"]`. The supervisor introduces finer-grained states. Migration strategy: add a `supervisor_status` column to the tasks table (`TEXT`, nullable) that holds the supervisor-specific state (`Pending`, `Assigned`, `Running`, `Completed`, `Failed`, `NeedsInput`). The existing `status` field maps as: `open` = Pending, `working` = Assigned|Running, `closed` = Completed|Failed|NeedsInput. This avoids breaking existing task CRUD while giving the supervisor the granularity it needs.

2. **Agent Health Monitor** (`racc-core/src/health.rs`)
   - PTY output analysis via a multi-signal approach (see "Completion & State Detection Strategy" below)
   - Per-agent-type pattern configs with concrete defaults:
     - **Claude Code**: completion = prompt reappearance (`╭─` or `$` after output); stuck = no output for 180s after last activity; error = lines matching `(?i)error|panic|fatal|SIGTERM`
     - **Aider**: completion = `>` prompt reappearance after output; stuck = no output for 120s after `>` prompt; error = Python traceback pattern
     - **Generic fallback**: completion = process exit; stuck = no output for 300s; error = exit code != 0
   - Heartbeat tracking: last output timestamp, output rate, rolling 60s output byte count
   - **PTY output subscription**: The supervisor taps into each session's existing `broadcast::Sender<Vec<u8>>` (`terminal_tx`) by calling `.subscribe()` to get a new `Receiver`. This is the same broadcast channel the frontend already subscribes to — adding a second subscriber does not interfere with UI rendering. The health monitor reads from this receiver in a dedicated async task per session.
   - **Lagged channel handling**: The broadcast channel has capacity 256. Under high-throughput agents (rapid output), the health monitor's receiver may hit `RecvError::Lagged(n)`. When this occurs, the monitor resets its idle timer (output clearly happened — silence detection would be a false positive) and continues. The 256 capacity may need tuning for agents producing very high output rates; this is a runtime config (`RACC_BROADCAST_CAPACITY`).

3. **Agent Input Adapter** (`racc-core/src/agent_adapter.rs`)
   - Translates task descriptions into agent-specific PTY input sequences:
     - **Claude Code**: writes the task description directly as a prompt string followed by `\n`
     - **Aider**: writes `/ask {task_description}\n` to use Aider's ask mode
     - **Codex**: passes task as a CLI argument at spawn time (`codex "{task_description}"`)
     - **Generic**: writes task description as stdin text followed by `\n`
   - Agent type is detected from the session's command string (e.g., contains "claude" → Claude Code adapter)
   - Adapters use the existing `Transport::write()` method to inject input into the PTY

4. **Task Scheduler** (`racc-core/src/scheduler.rs`)
   - Priority queue with optional dependencies
   - Agent affinity: prefer assigning related tasks to the same agent (same worktree)
   - Load balancing: distribute work across available agents

5. **Notification System** (extends existing `EventBus`)
   - In-app notifications (toast/badge in UI)
   - Optional webhook/desktop notification integration
   - Escalation levels: info, warning, needs-input, failure

6. **Supervisor Dashboard** (new React component)
   - Bird's-eye view: all tasks, their states, assigned agents
   - Health indicators: green/yellow/red per agent
   - One-click intervention: jump to agent terminal, retry task, reassign
   - Activity timeline: what happened while you were away

### Completion & State Detection Strategy

Completion detection is the hardest problem in this design. A multi-signal approach reduces false positives/negatives:

1. **Primary: PTY output pattern matching** — agent-specific regex patterns detect prompt reappearance, error output, and idle periods (see Health Monitor patterns above).
2. **Secondary: Process exit code** — if the agent process exits (`Transport::is_alive()` returns false), the task is done or crashed. Exit code 0 = completed, non-zero = failed.
3. **Tertiary: Git diff heuristic** — if the worktree has new commits since task assignment (checked via `git log --oneline {since_sha}..HEAD`), work likely happened. Combined with process exit or prompt reappearance, this confirms completion.
4. **Quaternary: Output volume analysis** — a sudden drop from active output to zero (with the process still alive) suggests either completion or stuck state. Cross-reference with pattern matching to disambiguate.

When signals conflict (e.g., process alive but no output and no prompt detected), the supervisor escalates to the user rather than guessing.

### Concurrency & Shared State

The existing DB uses `Mutex<Connection>` (sync). The supervisor loop introduces a new concurrent reader/writer. Strategy:

- Enable **SQLite WAL mode** — allows concurrent reads while one writer holds the lock. The supervisor's 5s reconciliation loop reads state (fast) and occasionally writes status updates (fast). WAL mode prevents read starvation.
- The supervisor acquires the DB lock for **short, bounded operations** only (read all task/session states, update a status field). No long-held locks.
- If the lock is contended (frontend CRUD in progress), the supervisor skips that reconciliation tick and retries on the next 5s interval. This is safe because the loop is periodic and idempotent.

### Worktree Lifecycle on Failure

When an agent fails a task (crash, error, or max retries exceeded):

1. **Keep the worktree and branch** — do not reset or delete. The partial work may contain useful progress or diagnostic output.
2. **Mark the worktree as `dirty`** in the DB — visible in the dashboard with a yellow/red indicator.
3. **User decides**: the dashboard offers three actions: (a) retry task in the same worktree, (b) discard changes and retry fresh, (c) inspect and manually resolve.
4. **Auto-cleanup policy**: dirty worktrees older than 7 days (configurable) are flagged for cleanup but never auto-deleted.

### Key Design Decisions

- **Graceful degradation**: If the supervisor can't determine agent state, escalate to user rather than take autonomous action. "I'm not sure if Agent #3 is stuck — want me to restart it?"
- **Supervisor is optional**: Users who prefer manual control can disable it. The existing interactive UX remains available.
- **State persistence**: Supervisor state lives in SQLite (existing DB). Survives app restarts.

### Implementation Phases

**Phase 1: Health Monitor + Dashboard (2-3 hours CC)**
- Add PTY output analysis to detect agent states
- Build supervisor dashboard component showing agent health
- No automation yet — just visibility

**Phase 2: Task Scheduler + Auto-Assignment (2-3 hours CC)**
- Task priority queue with state machine
- Auto-assign tasks to idle agents
- Basic agent pool management (spawn/kill)

**Phase 3: Reconciliation Loop + Self-Healing (2-3 hours CC)**
- Full supervisor reconciliation loop
- Stuck detection + auto-restart with backoff
- Escalation to user when uncertain
- Notification system

**Follow-up (separate design): Setup Simplification**
- One-command install, auto-config, first-run wizard — important but orthogonal to the supervisor pattern. Should be its own design doc.

## Open Questions

1. **Task description format**: Free text? Structured? How much context should a task carry for the supervisor to make good assignment decisions?
2. **Agent spawn limits**: Should there be a default cap on concurrent agents? (Resource protection for the server) Likely default: 5 concurrent agents.
3. **Worktree strategy**: One worktree per task? Per agent? Reuse worktrees across related tasks?

## Success Criteria

- A solo dev can define 5 tasks, walk away for 30 minutes, and come back to a clear summary of what completed, what failed, and what needs their attention
- Agent health status is visible at a glance (green/yellow/red)
- Self-healing works reliably for common failure modes (agent crash, network disconnect)
- Cognitive load scales sub-linearly with number of agents (managing 10 agents feels like managing 3)

## Dependencies

- Existing: PTY transport, session management, EventBus, SQLite DB, React UI
- New: No external dependencies expected — supervisor logic is pure Rust
- Nice-to-have: Desktop notification API (Tauri has this), webhook delivery
